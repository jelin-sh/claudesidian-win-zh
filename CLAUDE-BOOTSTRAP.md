# Obsidian 仓库指南 - Bootstrap 模板

**Claude Code + Obsidian 入门指南**

## 快速设置

1. **开始每个会话**: 运行 `git pull` 同步最新更改
2. **更改后**: 提交并推送以保存你的工作
3. **使用内置工具**: 优先使用 WebSearch 和 WebFetch 处理网页内容

## 版本控制最佳实践

**关键 - 每个会话开始**: 始终在每个新 Claude 会话开始时运行 `git pull`,以确保你拥有远程仓库的最新更改。

**提交流程**:

- 创建新笔记后: `git add .` → `git commit -m "message"` → `git push`
- 重要编辑后: 立即提交并推送
- 使用 `git status` 检查修改
- 当代理修改文件时: 始终提交这些更改

## 文件夹结构(PARA 方法)

```
vault/
├── 00_Inbox/           # 临时捕获点
├── 01_Projects/        # 有时间限制的倡议
├── 02_Areas/           # 持续的职责
├── 03_Resources/       # 参考资料
├── 04_Archive/         # 已完成/非活跃项目
├── 05_Attachments/     # 图片、PDF 等
│   └── Organized/      # 已处理的附件
└── 06_Metadata/        # 文档和模板
    ├── Reference/      # 指南和标准
    ├── Plans/          # 战略文档
    └── Templates/      # 可重用的结构
```

## PARA 方法详解

### Projects (项目 - 01)

- 有明确完成标准的时间限制倡议
- 示例: 撰写论文、开发演示文稿
- 推荐子文件夹: Research/(研究)、Drafts/(草稿)、References/(参考)、Output/(输出)

### Areas (领域 - 02)

- 没有结束日期的持续职责
- 示例: 健康、财务、职业发展
- 创建专门的笔记并链接到相关资源

### Resources (资源 - 03)

- 供参考的兴趣主题
- 按主题组织的知识库
- 用于不与特定项目关联的信息

### Archive (归档 - 04)

- 已完成或非活跃项目
- 保持与活跃部分相同的文件夹结构
- 定期审查以重新激活

## 收件箱管理

### 核心原则

- 收件箱是临时的,不是永久存储
- 使用捕获 → 处理 → 组织的工作流程每周处理
- 任何时候保持少于 20 个项目

### 保留在收件箱中的文件

- **关键**: 带数字前缀(00-06)的文件永久保留
- 最近的每日/每周摘要(最近 3 个月)
- 正在处理的活动笔记

### 处理工作流程

1. 删除过时信息
2. 将相关材料移动到 PARA 位置
3. 将行动转换为项目任务
4. 用 `#needs-processing` 标记需要更多处理的项目

## 文件组织指南

### 命名约定

- 每日笔记: `YYYY-MM-DD - 主题`
- 会议笔记: `Meeting - [主题] - YYYY-MM-DD`
- 想法: `Idea - [简短描述]`
- 资源: `Resource - [主题] - [来源]`

### 移动规则

- 使用 `mv` 命令(不是 `cp`)以避免重复
- 首先验证目标文件夹是否存在
- 移动后更新内部链接
- 组织时添加 YAML frontmatter

## 附件管理

### 组织

- 将所有非文本文件存储在 `05_Attachments/` 中
- 已处理的文件 → `05_Attachments/Organized/`
- 命名: `[相关笔记]_[描述].[扩展名]`

### 辅助脚本

```bash
pnpm attachments:list        # 列出未处理的文件
pnpm attachments:organized   # 统计已组织的文件
pnpm attachments:orphans     # 查找未被引用的文件
pnpm attachments:update-links # 移动后更新链接
```

## 网页内容工作流程

### 内置工具(首选)

- **WebSearch**: 用于一般网络搜索
- **WebFetch**: 用于特定 URL
- 根据内容类型保存到适当的文件夹

### 自定义脚本(需要时)

- 单个 URL: `pnpm firecrawl:scrape <url> <output>`
- 批量 URL: `pnpm firecrawl:batch <url1> <url2>`
- 保存到 `00_Inbox/Clippings/` 并添加 frontmatter

## 写作风格指南

### 结构

- 使用 `[[WikiLinks]]` 进行内部引用
- 包含 YAML frontmatter(日期、标签、状态)
- 一致的 Markdown 格式
- 具体、一致的标签

### 风格偏好

- 直接和自信的陈述
- 避免陈词滥调的过渡
- 让陈述独立存在
- 没有不必要的引导词

## AI 助手指南

### 任何组织之前

1. 映射完整的文件夹结构: `find . -type d | sort`
2. 在 `06_Metadata/STRUCTURE.md` 中记录
3. 验证所有目标文件夹都存在

### 处理内容

- 尊重编号的核心文件(永不移动 00-06 前缀的文件)
- 组织时始终使用 `mv` 而不是 `cp`
- 保留并更新双向链接
- 添加适当的 YAML frontmatter

### 仅使用简单命令

- **必需**: 直接、基本的命令,无需过滤
- **禁止**: 复杂的正则表达式、管道命令、带过滤器的 find
- 正确示例: `ls -1` 然后手动选择文件
- 错误示例: `ls | grep pattern` 或 `find . -name "*.png"`

## 每日工作流程

### 一天开始时

1. 运行 `git pull`
2. 检查收件箱中要处理的项目
3. 审查活跃项目

### 一天结束时

1. 处理新的收件箱项目
2. 提交并推送更改
3. 更新项目笔记

### 每周审查

1. 处理整个收件箱
2. 归档已完成的项目
3. 更新领域笔记
4. 审查和整合资源

## 项目生命周期

### 启动项目

1. 在 `01_Projects/[项目名称]` 中创建文件夹
2. 添加子文件夹: Research/(研究)、Drafts/(草稿)、Output/(输出)
3. 创建包含目标和时间线的 README

### 项目进行中

- 将所有相关材料保留在项目文件夹中
- 链接到相关资源和领域
- 定期提交以跟踪进度

### 完成项目

1. 创建项目摘要笔记
2. 将整个文件夹移动到 `04_Archive/`
3. 更新相关领域笔记
4. 使用完成消息提交

## 最佳实践

### 组织

- 保持文件夹结构浅层(最多 3 层)
- 只有 7 个以上相关笔记时才创建子文件夹
- 使用链接而不是深度嵌套
- 在主要文件夹中包含 README

### 内容创建

- 先捕获,后组织
- 每个笔记一个想法
- 大量使用链接
- 一致地标记

### 维护

- 每周处理收件箱
- 每月审查项目
- 每季度清理归档
- 定期 git 提交
